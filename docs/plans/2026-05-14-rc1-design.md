# RC1 Design — Split-App Architecture

_Brainstormed 2026-05-14._

## Goal

Ship the daily-flower app to **TestFlight + Internal Play Store closed beta** as RC1, on a one-month timeline, while standing up a **separate sibling project** for the rich iridescent specimen-card experience instead of layering it onto the daily-flower app again.

## Why split

Three days of recent history show the cost of layering heavy native deps onto this codebase:

- `cfce8e3` reverted the entire IridescentCard component family because it "broke" — though it worked on web, EAS native builds failed
- `94957e1` ripped out `@shopify/react-native-skia`, `react-native-svg`, `expo-sensors`
- `2838dd7` removed `react-native-gesture-handler` because it was "causing native build failures"
- `8e8d9e2` disabled `libdav1d` in `expo-image` to unblock EAS

Each revert was the right move for stability, but the pattern shows the daily-flower app and the iridescent experience are fighting each other. A clean perforation between two apps means each stays simple internally and we never have to revert visual work to recover ship-ability.

## The two apps

| | **App A** | **App B** |
|---|---|---|
| **What it is** | The daily-flower app, current state | A new sibling project that owns the iridescent specimen-card experience |
| **Tech baseline** | Stable: FlipCard, no Skia, no gesture-handler, no sensors, no extra native deps | Heavy: Skia + react-native-svg + expo-sensors + react-native-gesture-handler + IridescentCard component family |
| **Identity** | Rebrand mixing _Playing in the Sandbox_ × _Staring at Flowers_ themes (see Phase 2) | Working title TBD, separate identity |
| **In RC1 month** | **Ship to TestFlight + Internal Play Store** | **Scaffold + port + native build green on a real device** (no store submission this month) |
| **Code location** | This repo at `~/workspace/playing-in-the-sandbox-looking-at-flowers` | New sibling repo (e.g. `~/workspace/appB`) |
| **Source of IridescentCard** | n/a — archived plan docs preserved at `docs/plans/archive/` | Ported from git history (`f0e968a` and earlier commits on this repo) |

## Approach

**Approach C — incremental integration**: each phase delivered in small PRs merged to main as ready. Daily-buildable, daily-demoable. Phase order is linear because some tracks unblock others.

This pattern matched what worked best in the last three days. The 19-commit PR from `claude/practical-swartz-890af7` was too big to easily un-break when one dep caused EAS failures; smaller PRs would have isolated the regression.

## App A — phase plan

### Phase 0 — Native unlock (prerequisite)

User-action: install iOS 26.5 simulator runtime via `xcodebuild -downloadPlatform iOS`.

Then baseline both platforms:
- `eas build --profile development --platform ios --local`
- `eas build --profile development --platform android --local`

Both should be green against current main; the heavy deps that caused trouble are already out. Capture the working flow in `meta/DEVELOPMENT.md`.

### Phase 1 — Cleanup

- Move `docs/plans/2026-05-13-iridescent-card-design.md` and `docs/plans/2026-05-13-iridescent-card.md` to `docs/plans/archive/` (historical record + source for App B's port)
- Audit the 14 files in `meta/` — prune anything stale, consolidate duplicates
- Local-only: delete `.playwright-cli/` debug logs

### Phase 2 — Rebrand (no website, no domain)

App-only rebrand. Mix both phrase themes (_Playing in the Sandbox_ + _Staring at Flowers_).

1. **Name workshop** — generate 8-12 candidates against a positioning brief (one sentence from user). Examples for vibe only: _Specimen Sandbox_ (matches the foil-stamped card metaphor), _Sandbox & Bloom_, _Field Studies_, _Petals in the Sandbox_. Compressed form of current name (_Sandbox Staring at Flowers_) also valid.
2. **Icon** — vector source in `app_store_assets/icons/`. 1024 master + Android adaptive (foreground + background). Built from a brand mark so resizing is clean.
3. **Brand tokens** — new file `constants/brand.ts` consolidates colors + typography. Current values are scattered across `app/index.tsx` and `app/flower-detail.tsx`.
4. **Voice doc** — `meta/VOICE.md`. Sets register for in-app copy, error states, accessibility labels, and any future store listings.
5. **`app.json` updates** — name + icon + splash. **Slug unchanged** (changing it would reset the EAS Update URL — deferred to RC2).

### Phase 3 — Hardening: errors + offline (parallel with Phase 2)

The only hardening dimension in scope. Accessibility, performance, telemetry are deferred to RC2.

1. **Error path audit** — inventory every place the app can fail (fetch, region, image decode, navigation), documented in `meta/ERROR_STATES.md`
2. **Offline detection** — `@react-native-community/netinfo` listener
3. **Bundled fallback flowers** — `assets/defaults/` already has sunrise/midday/sunset; wire as last-resort fallbacks; reconcile inconsistent `getDefaultFlower()` usage in `lib/dailyFlower.ts`
4. **Retry strategy** — exponential backoff on 5xx, no retry on 404
5. **Acceptance**: airplane-mode → "OFFLINE · ARCHIVE" badge over fallback flower; reconnect refreshes

### Phase 4 — Dev env + docs + workflows (Expo-skills-informed)

| Expo skill | Applied to |
|---|---|
| `expo-deployment` | EAS profiles audit, credentials check, TestFlight + Internal Play submission walkthrough |
| `expo-dev-client` | `pnpm exec eas build --profile development` dev loop documented |
| `building-native-ui` | HIG audit of FlipCard flip + flower-detail close + nav row |
| `expo-cicd-workflows` | `.github/workflows/ci.yml` tightening (PR triggers, lint, possibly Maestro) |
| `upgrading-expo` | Codify the SDK 55 patch-bump pattern (already used multiple times) |
| `eas-update-insights` | OTA strategy: do beta builds receive instant OTA fixes, or cut new builds? |

Deliverables:
- `README.md` refresh (currently ~5 KB, likely stale)
- `meta/DEVELOPMENT.md` refresh
- `meta/DEPLOYMENT.md` (new — TestFlight + Play Internal walkthrough)
- `.github/workflows/ci.yml` upgrade

### Phase 5 — RC1 cut

1. Tag `appA-rc1` on the merge tip
2. `eas build --profile production --platform ios` → submit to TestFlight
3. `eas build --profile production --platform android` → upload to Play Console internal track
4. Tester invites (user supplies tester emails / Play group)
5. **Acceptance**: ≥1 external tester installs both, 24h crash-free use

## App B — single-phase scaffold

After App A ships (or in parallel late in the month). App B's full RC1 is a future plan, not this one.

1. `npx create-expo-app appB --template default` in `~/workspace/` (sibling to this repo)
2. `pnpm exec expo install @shopify/react-native-skia react-native-svg expo-sensors react-native-gesture-handler` (SDK 55–matched)
3. Port `components/IridescentCard/*` from `f0e968a` (reverted on App A but still in git history) — cherry-pick or manual copy
4. Minimal `app/index.tsx` that just renders `<IridescentCard>` with sample data (no daily-flower fetch logic — App B will define its own data model later)
5. **Solve the EAS native build with full Skia stack.** Debugging-first: diagnose what broke in App A (libdav1d clash? gesture-handler version pin? stale prebuild artifacts?) before reintroducing dep-by-dep
6. **Acceptance**: iridescent card visible on a physical iPhone via TestFlight dev profile, no crashes for 5 minutes

## Risks

| Risk | Mitigation |
|---|---|
| Native build still fails after iOS 26.5 install | Current main has no heavy deps — should be green. If not, debug as Phase 0 sub-task, don't move on. |
| Rebrand expands beyond app-only scope (website, domain) | Explicitly out of scope; revisit after RC1 |
| App B scaffold takes longer than estimated | App B doesn't gate App A's ship date. Defer to RC2 if it overruns. |
| EAS CI minute quota (hit during PR #12) | Local EAS builds for verification; cloud only for submission |
| TestFlight review delays | iOS review can take 1-3 days; build early in the final week, not on the last day |

## Out of scope for RC1

- App B store submission (its own future RC1)
- Public website / domain / marketing surface
- Accessibility audit beyond what we already have (`accessibilityLabel` on cards/buttons)
- Performance instrumentation / telemetry
- Crash reporting wiring (Sentry/Bugsnag)
- Test coverage beyond `pnpm typecheck` and existing Maestro
- SDK upgrades beyond patch versions on SDK 55
